# 为什么 AI 写代码更快但交付没变，以及我怎么把它扳回来的

## 一句话总结

AI 加速了编码环节，但大多数团队的 SDLC 瓶颈在 review 不在 coding——上游加速产生的增量被下游瓶颈原封不动吃掉，导致个体感知变快但组织交付速度纹丝不动。破解之道不是选更好的工具，而是重构 SDLC 结构：画清 Spec/Rule/Skill 职责边界 → 搭建验证闭环 → 引入并发 → 把省出的时间投入能力进化。

---

## 核心论点

### 1. 主观快 20%，客观慢 19%

METR（预注册 RCT 实验，16 名资深开源开发者）：AI 辅助编码后客观测量完成时间**慢 19%**，但开发者自己觉得**快 20%**。主客观偏差达 39 个百分点。

Faros AI（10,000+ 开发者遥测数据）：高 AI 采纳团队个体指标全面飘红——任务完成数 +21%，PR 合并数 +98%。但 DORA 四大交付指标**无一改善**。

根因：PR 体积 +154%，评审时间 +91%。AI 帮你写了两倍的代码扔给同一个 reviewer，他的带宽没有翻倍。

### 2. 约束理论：非瓶颈加速导致下游堆积

1984 年 Goldratt《目标》中的工厂案例：花重金引进高效机器人 → 该工位效率提升 36% → 整个工厂交付周期和利润**恶化**。因为瓶颈不在那个工位，在热处理炉。

**AI 就是今天的 NCX-10**。它把 coding 效率拉到极致，但 SDLC 的瓶颈在需求理解、兼容性分析、Code Review、测试验证。加速非瓶颈环节，唯一的效果是下游堆积。

### 3. 框架选型不重要，Spec/Rule/Skill 区分才重要

OpenSpec、Superpowers、BMAD、Spec Kit 本质在做同一件事：将 SDLC 中重复度高的 prompt 沉淀为可复用 Skill。差异是风格偏好，不是对错。

三者本质区别：

| 维度 | Spec | Rule | Skill |
|---|---|---|---|
| 本质 | 业务知识 | 工程规范 | 工作流 |
| 回答的问题 | 做什么 | 约束是什么 | 怎么一步步做 |
| 加载方式 | 被 Skill 消费 | 始终在上下文头部 | 按需加载到尾部 |
| 长度 | 视复杂度 | 短，300-500 行上限 | 可长，不受限 |

**关键洞察：区别不在语义标签，在加载机制。** Rule 始终驻留头部，Skill 按需加载到尾部（近因效应），Spec 作为 Skill 的输入数据被注入。

Rule 文件写到 1000 行 → 大量内容落入注意力薄弱区 → 模型开始"忘记"中间规则。Skill 按需加载 + 高度聚焦，信噪比从 3/50 变成接近 1:1。

**Skill 的三个能力维度：** SOP 流程指令 + Few-shot 示例（Reference 目录）+ CLI/脚本集成（verify 类 Skill 直接调用 `npm run test`）。

### 4. 安灯绳：验证闭环是质量的底线

AI 生成代码的问题率（10.83/个）是人工作业（6.45/个）的 1.7 倍。但 48% 的开发者"总是"在提交前检查 AI 代码——超过一半的人扫一眼就提交了。

**给 AI 装安灯绳**：Lint + Unit Test + E2E 验证。generate → verify → fix/log 闭环。验证不是万能的，但把低级错误从人工发现变成自动发现，体感质量提升巨大。

瑞士奶酪模型：Lint 拦语法、Review 拦逻辑、Unit Test 拦边界、E2E 拦流程断裂。四层叠加后穿透率极低。信任链给机制，不给人。

### 5. 单任务变慢不是问题，你不能并发才是

按完整 SDLC 走（explore → design → implement → verify），单任务比直接手写慢 20%-100%（METR 测出慢 19%）。但这个"慢"换来的是返工率大幅下降。

**并发逻辑**：你能在 review 任务 A 时，让任务 B/C/D 在自动生成和自验证。可复制性是 AI 真正的优势。可行并发度约 2-3（Anthropic 2026.1 数据：可完全托管任务占比仅 0-20%）。

**但并发有前提：验证闭环必须先建好**。没有自动验证，你得逐个打开代码读，并发的意义归零。

并发要配合 WIP（在制品）限制：控制同时处于 review 阶段的 PR 数量和体积。不限制在制品数量的并发，本质是把 NCX-10 问题从一台变成五台。

### 6. 洗衣机悖论：省出的时间在洗更多衣服

美国家庭添置洗衣机后，家务时间没有减少——技术提升了效率，也提升了标准：以前一周洗一次，现在穿一次就洗。

映射到 AI Coding：苦差事占比稳定在每周 23%-25%（SonarSource），但成分变了：从"手写样板代码"变成"管理技术债"和"返工 AI 代码"。ATM 让银行柜员数量反而增长，但工作内容从数钱变成了销售理财。

GitClear 数据：Copy/Paste 代码首次超过 Moved code；代码搅动率上升 84%；commit comments 下降 27%。AI 在帮你复制粘贴，不在帮你思考架构。

**正确的用法：洗衣机洗衣服，你去读书。** 用省出的时间做以前做不到的事（多 Agent 并行拆解 10 份行业报告、精读论文），这是能力进化，不是效率提升。

### 7. 保底行动框架

**风险分级：**
- **第一档（货币相关）**：别让 AI 独立完成。一次出错代价大于省的所有时间。
- **第二档（容错可重试）**：文档、技术调研、测试用例初稿。best-of-N 筛选质量。
- **第三档（重复性高）**：自动化 Git Commit、自动化日报。每天省 10 分钟 × 200 工作日 = 33 小时/年 = 精读 5-6 篇论文。

## 关键引用

- METR RCT：16 资深开发者，预注册随机对照，AI 辅助编码客观慢 19%，主观快 20%
- Faros AI：10,000+ 开发者，高采纳团队任务 +21%，PR +98%，DORA 指标零改善；PR 体积 +154%，评审时间 +91%
- DORA/Stack Overflow/JetBrains：三份调查交叉验证，90% 在用 AI，7% 总是使用，3.1% 高度信任，不信任率 46% 首次超信任率 33%
- Liu et al. 2024 (TACL)：LLM 上下文利用呈 U 型曲线，中间位置性能下降超 20%
- Tian et al. 2025 (ACL 2025, LongPiBench)：相对位置偏差仍导致 20%-30% 性能波动，scale up 无法解决
- SonarSource：每周苦差事稳定在 23%-25%，不因 AI 使用频率变化
- GitClear：AI 生成 PR 平均问题 10.83 个 vs 人工 6.45 个，差距 1.7 倍
